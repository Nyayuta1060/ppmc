# ppmc 設計書

## 1. 概要

`ppmc` は Rust で実装する、PDF スライド向けのプレゼンテーションツールである。  
`pdfpc` のように、聴衆向け画面と発表者向け画面を分け、現在スライド、次スライド、発表者ノート、タイマー、進行状況を扱えることを目標にする。

特に重視する特徴は、PC とスマートフォンを OS に依存しない形で連携させ、スマートフォンからプレゼン画面を操作できることである。iOS / Android のネイティブアプリに依存せず、スマートフォンの標準ブラウザから利用できる Web コンソール方式を基本方針とする。

プロジェクト名 `ppmc` は `pdf-presenter-mobile-console` の略称である。PDF presenter と mobile console を一体で扱う意図を表す。

## 2. 目的

- PDF をそのままプレゼン資料として表示できる。
- 発表者は別画面で現在スライド、次スライド、ノート、経過時間を確認できる。
- キーボード、マウス、外部デバイス、モバイル端末から操作できる。
- iOS / Android / PC OS に縛られないブラウザベースのモバイル連携を提供する。
- Rust 製のデスクトップアプリとして配布しやすくする。
- 小さな MVP から始め、高度なモバイル連携や追加フォーマットに拡張できる構成にする。

## 3. 想定ユーザー

- PDF スライドで発表する開発者、研究者、講師。
- Linux 環境で軽量なプレゼンツールを使いたいユーザー。
- 手元のスマートフォンやタブレットを発表者コンソールとして使いたいユーザー。

## 4. 対象プラットフォーム

### MVP

- Linux デスクトップ環境。
- X11 または Wayland 上で動作する GUI アプリケーション。
- 1 画面構成でも利用可能。
- 2 画面構成では、片方を聴衆画面、もう片方を発表者画面として使う。

### 将来対応候補

- macOS。
- Windows。
- ネットワーク越しの複数端末連携。
- オフライン利用しやすい PWA 化。

## 5. MVP スコープ

最初のリリースでは、以下を実装対象とする。

- PDF ファイルを読み込む。
- スライドを 1 ページ単位で表示する。
- 前後のページへ移動する。
- 聴衆画面と発表者画面を表示する。
- 発表者画面に以下を表示する。
  - 現在スライド。
  - 次スライド。
  - 現在ページ番号と総ページ数。
  - 経過時間。
  - 現在時刻。
- キーボード操作を提供する。
- スマートフォンのブラウザからページ送り、ページ戻し、タイマー操作を行える。
- スマートフォン接続用の URL と QR コードを表示する。
- 基本的な設定ファイルを読み込む。
- PDF のレンダリング結果をキャッシュする。現在は表示ページ単位の簡易メモリキャッシュから実装を開始している。

MVP では以下を対象外とする。

- アニメーション付き PDF の段階表示。
- PDF 内動画、音声の再生。
- 複数スマートフォンの同時操作。
- 発表者ノートの高度な編集。
- クラウド同期。

## 6. ユースケース

### 6.1 通常プレゼン

1. ユーザーが `ppmc slides.pdf` を起動する。
2. アプリケーションが PDF を読み込む。
3. 聴衆画面に 1 ページ目を表示する。
4. 発表者画面に現在スライド、次スライド、タイマーを表示する。
5. ユーザーがキーボードまたはクリックでスライドを進める。

### 6.2 1 画面での確認

1. ユーザーが `ppmc --single slides.pdf` を起動する。
2. 1 つのウィンドウに発表者画面を表示する。
3. ユーザーは発表練習や資料確認に使う。

### 6.3 発表者ノート付きプレゼン

1. ユーザーが PDF とノート情報を用意する。
2. アプリケーションがページ番号に対応するノートを読み込む。
3. 発表者画面に現在ページのノートを表示する。

### 6.4 スマートフォン連携プレゼン

1. ユーザーが PC で `ppmc --remote slides.pdf` を起動する。
2. アプリケーションがローカル Web コンソールを起動する。
3. 発表者画面に接続用 URL と QR コードを表示する。
4. ユーザーがスマートフォンのブラウザで URL を開く。
5. スマートフォンから次ページ、前ページ、タイマー操作を行う。
6. PC 側の聴衆画面と発表者画面が操作に追従する。

## 7. 画面仕様

### 7.1 聴衆画面

聴衆画面はスライド表示に集中する。

- PDF ページをウィンドウまたはディスプレイ全体に表示する。
- アスペクト比を維持する。
- 背景は黒を基本とする。
- ページ外の余白は黒で埋める。
- マウスカーソルは一定時間操作がなければ非表示にする。

### 7.2 発表者画面

発表者画面は操作と確認に必要な情報を密度高く表示する。

- メイン領域: 現在スライド。
- サブ領域: 次スライド。
- 情報領域:
  - 現在ページ / 総ページ数。
  - 経過時間。
  - 現在時刻。
  - 必要なら残り時間。
- ノート領域:
  - 現在ページに対応する発表者ノート。
  - 長いノートはスクロール可能にする。

レイアウトは 16:9 の横長ディスプレイを基準にし、狭い画面では現在スライドとノートを優先する。

## 8. 操作仕様

### 8.1 キーボード

| 操作 | キー |
| --- | --- |
| 次ページ | `Right`, `Down`, `Space`, `PageDown`, `n` |
| 前ページ | `Left`, `Up`, `Backspace`, `PageUp`, `p` |
| 先頭ページ | `Home` |
| 最終ページ | `End` |
| フルスクリーン切り替え | `f` |
| タイマー開始 / 一時停止 | `t` |
| タイマーリセット | `r` |
| 終了 | `q`, `Esc` |

キー割り当ては設定ファイルで変更可能にする。

### 8.2 マウス / ポインタ

- 左クリックで次ページへ進む。
- 右クリックで前ページへ戻る。
- ホイール下で次ページ、ホイール上で前ページへ移動する。
- 将来的にレーザーポインタ風の注釈表示を追加できるよう、ポインタ位置はアプリケーション状態として扱う。

### 8.3 リモート操作

リモート操作は本プロジェクトの中核機能として扱う。MVP では、OS に依存しないスマートフォン連携を実現するため、PC 上にローカル Web サーバーを起動し、スマートフォンのブラウザから操作する方式を採用する。

- ローカル HTTP/WebSocket サーバーを起動する。
- モバイルブラウザから接続する。
- 次ページ、前ページ、タイマー操作を送信する。
- 発表者画面の簡易情報をモバイル端末に表示する。
- iOS / Android のネイティブアプリは必須にしない。
- 接続用 URL と QR コードを PC 側に表示する。
- 同一 LAN 内での利用を基本とする。
- 認証なしの操作を避けるため、起動ごとのランダムトークンを URL に含める。

### 8.4 モバイル Web コンソール

スマートフォン側の UI は、ブラウザで動く軽量な Web アプリケーションとする。

MVP では「操作 + 状態表示」を範囲とし、発表中に誤操作しにくい大きな操作面を最優先する。

- 大きな `次へ` / `戻る` ボタンを表示する。
- 現在ページ / 総ページ数を表示する。
- 経過時間と現在時刻を表示する。
- タイマー開始、一時停止、リセットを操作できる。
- 通信状態を表示する。
- 画面スリープを抑制できる場合は Wake Lock API を利用する。
- JavaScript が利用できるモダンブラウザを前提とする。

早期拡張として、以下を検討する。

- `操作` / `ノート` / `プレビュー` のタブ構成。
- 発表者ノート表示。
- 現在スライドまたは次スライドのサムネイル表示。
- PWA としてホーム画面に追加できる形式。
- PC とスマートフォン間の接続再開。

## 9. 入力ファイル仕様

### 9.1 PDF

- 1 ページを 1 スライドとして扱う。
- ページサイズが混在する PDF も許容する。
- ページ番号は 1 始まりで表示し、内部状態は 0 始まりで扱う。
- GUI ではファイルダイアログからの PDF 選択を基本とする。
- 開発・デバッグ用途として PDF パスの直接入力も許容する。

### 9.2 発表者ノート

MVP では外部ファイル方式を採用する。ファイル拡張子は `.ppmc` とし、PDF と同じベース名のサイドカーファイルを自動検出する。例: `slides.pdf` に対して `slides.ppmc`。

`.ppmc` は TOML を基本とし、まずはページ単位の発表者ノートから実装する。将来的にタイトル、予定時間、ページ単位設定、モバイル表示設定を同じファイルへ拡張する。

```toml
version = 1

[pages.1]
notes = """
1 ページ目のノート
"""

[pages.2]
notes = """
2 ページ目のノート
"""
```

PDF 内のノート抽出と `.pdfpc` 互換インポートは将来対応とする。

## 10. 設定仕様

設定ファイルは TOML を基本とする。

設定ファイル探索順:

1. CLI オプション `--config` で指定されたパス。
2. カレントディレクトリの `ppmc.toml`。
3. ユーザー設定ディレクトリの `ppmc/config.toml`。
4. 組み込みデフォルト。

設定例:

```toml
[window]
audience_fullscreen = true
presenter_fullscreen = false
background = "#000000"

[timer]
auto_start = false
planned_minutes = 20

[notes]
path = "slides.ppmc"

[keys]
next = ["Right", "Down", "Space", "PageDown", "n"]
previous = ["Left", "Up", "Backspace", "PageUp", "p"]
quit = ["q", "Esc"]
```

## 11. CLI 仕様

基本形式:

```bash
ppmc [OPTIONS] <PDF_PATH>
```

主なオプション:

| オプション | 説明 |
| --- | --- |
| `--config <PATH>` | 設定ファイルを指定する |
| `--notes <PATH>` | ノートファイルを指定する |
| `--single` | 1 ウィンドウの発表者画面として起動する |
| `--audience-display <INDEX>` | 聴衆画面に使うディスプレイを指定する |
| `--presenter-display <INDEX>` | 発表者画面に使うディスプレイを指定する |
| `--start-page <PAGE>` | 開始ページを 1 始まりで指定する |
| `--duration <MINUTES>` | 発表予定時間を分単位で指定する |
| `--remote` | モバイル連携用のリモート操作サーバーを起動する |
| `--remote-port <PORT>` | リモート操作サーバーのポートを指定する |
| `--remote-host <ADDR>` | リモート操作サーバーの待ち受けアドレスを指定する |

インストール後は、配布形式に関係なく `ppmc slides.pdf` で起動できることを基本 UX とする。

## 12. アーキテクチャ

### 12.1 モジュール構成

```text
ppmc
├── app          # アプリケーション状態とイベント処理
├── cli          # CLI 引数の解析
├── config       # 設定ファイル読み込みとデフォルト値
├── document     # PDF とノートの読み込み
├── render       # PDF レンダリングとキャッシュ
├── ui           # Tauri ウィンドウ制御と GUI 表示
├── frontend     # PC 画面とモバイル Web コンソールの Web UI
├── input        # キーボード、マウス、リモート入力の正規化
├── remote       # HTTP/WebSocket によるモバイル連携
└── time         # タイマー、現在時刻、予定時間
```

### 12.2 状態モデル

中心状態は `PresentationState` として管理する。

```rust
struct PresentationState {
    current_page: usize,
    total_pages: usize,
    timer: TimerState,
    pointer: Option<PointerState>,
    mode: PresentationMode,
}
```

入力イベントは UI 固有のイベントからアプリケーション内部の `Command` に変換する。

```rust
enum Command {
    NextPage,
    PreviousPage,
    GoToPage(usize),
    ToggleFullscreen,
    ToggleTimer,
    ResetTimer,
    Quit,
}
```

この方針により、キーボード、マウス、リモート操作を同じ経路で扱う。

### 12.3 モバイル連携アーキテクチャ

PC 側の `remote` モジュールは、ローカルネットワーク向けに HTTP と WebSocket を提供する。

- HTTP:
  - モバイル Web コンソールの HTML / CSS / JavaScript を配信する。
  - 接続確認用のヘルスチェックを提供する。
- WebSocket:
  - スマートフォンから PC へ操作コマンドを送る。
  - PC からスマートフォンへ現在ページ、総ページ数、タイマー状態を送る。
  - 接続切断と再接続を扱う。

通信メッセージは JSON を基本とする。

```json
{
  "type": "command",
  "command": "next_page"
}
```

```json
{
  "type": "state",
  "current_page": 3,
  "total_pages": 42,
  "timer_seconds": 180,
  "timer_running": true
}
```

セキュリティ方針:

- デフォルトでは LAN から接続できるアドレスを選択できるようにする。
- 起動時にランダムな接続トークンを生成する。
- トークンなしの操作 API は受け付けない。
- インターネット公開は想定しない。
- HTTPS は MVP では必須にしないが、将来のオプションとして扱う。

### 12.4 Tauri GUI アーキテクチャ

PC 側 GUI は Tauri を第一候補とする。Rust 側にアプリケーション状態、PDF レンダリング、リモート操作サーバーを置き、Tauri の WebView では UI 表示と入力受付を担当する。

- 聴衆画面と発表者画面は Tauri の複数ウィンドウとして扱う。
- 発表者画面、聴衆画面、モバイル Web コンソールは同じ Web 技術で実装する。
- PDF ページは WebView の PDF ビューアに直接表示させず、Rust 側でレンダリングした画像を表示する。
- Rust 側と UI 側は Tauri command / event、またはローカル API 経由で状態を同期する。
- フルスクリーン、マルチディスプレイ、カーソル非表示は Tauri のウィンドウ API で制御する。

注意点:

- Linux では WebKitGTK などの WebView 実行環境に依存する。
- マルチディスプレイとフルスクリーン挙動はデスクトップ環境差が出やすいため、初期に小さな検証アプリを作る。
- 完全な単体バイナリ配布より、Tauri の標準的なパッケージ配布を優先する。

## 13. 技術選定

### 13.1 言語

- Rust stable。

### 13.2 GUI 候補

候補:

- Tauri。
- `egui` / `eframe`。
- `winit` + `wgpu`。
- `iced`。

初期候補は Tauri とする。理由は、PC 側の発表者画面とスマートフォン向け Web コンソールを同じ Web 技術で作りやすく、Rust バックエンドと組み合わせられるため。

Tauri を採用する場合も、PDF レンダリングそのものは WebView に任せない。Rust 側で PDF ページを画像化し、Web UI は画像表示、レイアウト、操作受付を担当する。

Tauri でマルチディスプレイ、フルスクリーン、低遅延なページ切り替えに問題が出る場合は、`winit` + `wgpu` を再検討する。

### 13.3 PDF レンダリング候補

候補:

- `pdfium-render`。
- Poppler バインディング。
- WebView 組み込み PDF ビューア。

初期候補は `pdfium-render` とする。理由は Rust から扱いやすく、ページを画像としてレンダリングする用途に合いやすいため。

WebView 組み込み PDF ビューアは、ページ単位の制御、キャッシュ、サムネイル生成、モバイル連携との同期が難しくなるため採用しない。

配布方式は、リリースでは PDFium をアプリに同梱する方針を第一候補とする。開発時は同梱 PDFium が見つからない場合にシステムライブラリへフォールバックできるようにする。

### 13.4 設定 / CLI

- デスクトップ GUI: Tauri。
- フロントエンド: TypeScript + Vite を第一候補。
- CLI: `clap`。
- 設定: `serde` + `toml`。
- モバイル連携: `axum` または `tokio` + WebSocket 対応クレート。
- QR コード生成: `qrcode`。
- ログ: `tracing`。
- エラー: `anyhow` または `thiserror`。

ライブラリ境界が明確になった段階で、アプリケーション層は `anyhow`、ドメイン寄りのエラーは `thiserror` に分ける。

## 14. レンダリング設計

PDF ページは必要な解像度でラスタライズし、UI 側ではテクスチャとして表示する。

- 現在の実装では、PDF パス、ページ番号、レンダリング幅、最大高さをキーにしたメモリキャッシュを持つ。
- PDF 読み込み時に既存キャッシュをクリアする。
- 次の段階で現在ページと次ページを優先的にレンダリングする。
- 前後数ページをバックグラウンドで先読みする。
- 表示倍率やディスプレイ DPI に応じたキャッシュキーを持つ。
- メモリ上限を設け、古いページから破棄する。

キャッシュキー例:

```rust
struct PageCacheKey {
    pdf_path: String,
    page_index: usize,
    width: i32,
    max_height: i32,
}
```

## 15. エラー処理

ユーザーに表示するべきエラー:

- PDF ファイルが存在しない。
- PDF を開けない。
- PDF レンダリングに失敗した。
- ノートファイルの構文が不正。
- 設定ファイルの構文が不正。
- 指定されたディスプレイが存在しない。
- リモート操作ポートを開けない。
- モバイル連携の接続トークンが不正。

基本方針:

- CLI 起動時に検出できる問題は標準エラーへ明確に出す。
- GUI 起動後の問題は発表者画面にエラーバナーとして表示する。
- 聴衆画面には可能な限り内部エラーを出さない。

## 16. テスト方針

### 単体テスト

- CLI 引数の解析。
- 設定ファイルのマージ。
- キー割り当てから `Command` への変換。
- ページ遷移ロジック。
- ノートファイルのパース。
- タイマー状態遷移。
- リモート操作メッセージから `Command` への変換。
- 接続トークン検証。

### 結合テスト

- サンプル PDF を読み込んでページ数を取得できる。
- 先頭、末尾、範囲外ページ移動の挙動。
- 設定ファイルと CLI オプションの優先順位。
- WebSocket 経由で次ページ、前ページコマンドを受け取れる。

### 手動確認

- 1 画面モード。
- 2 画面モード。
- フルスクリーン切り替え。
- PDF 表示品質。
- 長時間起動時のメモリ使用量。

## 17. フェーズ分割

### Phase 1: 最小ビューア

- Rust プロジェクトの初期化。
- CLI で PDF パスを受け取る。
- PDF を読み込んで 1 ページを表示する。
- 前後ページ移動。

### Phase 2: 発表者画面

- 現在スライドと次スライドの表示。
- ページ番号、現在時刻、経過時間。
- 1 画面モードと 2 画面モード。

### Phase 3: モバイル連携

- ローカル HTTP/WebSocket サーバー。
- スマートフォン向け Web コンソール。
- 接続用 URL と QR コード表示。
- 次ページ、前ページ、タイマー操作。
- 現在ページ、総ページ数、経過時間、通信状態の表示。
- 起動ごとの接続トークン。

### Phase 4: ノートと設定

- `.ppmc` ノートファイル読み込み。

## 18. 配布方針

初期の公開先は GitHub Releases とする。リリース成果物は GitHub Actions でビルドし、タグに紐づけて配布する。

### 18.1 Linux 配布形式

優先順位は以下とする。

1. AppImage。
2. deb パッケージ。
3. `curl` / `wget` で取得できるインストールスクリプト。
4. snap は後続対応候補。

AppImage はディストリビューション差を吸収しやすいため、最初の汎用 Linux 配布形式として優先する。deb は Debian / Ubuntu 系ユーザー向けに提供する。

配布形式に関係なく、インストール後の起動コマンドは `ppmc` に統一する。deb では `/usr/bin/ppmc` を提供する。AppImage では、インストールスクリプトが AppImage 本体を `~/.local/share/ppmc/` などに配置し、`~/.local/bin/ppmc` に wrapper または symlink を作成する。

`curl` / `wget` 方式は、GitHub Releases の最新版を取得してユーザーのローカルに配置する薄いインストールスクリプトとして設計する。スクリプトは配布物のチェックサム検証を行う。

GitHub Releases の Linux x86_64 asset は以下の命名規則にする。

- `ppmc-$TAG-linux-x86_64.AppImage`
- `ppmc-$TAG-linux-x86_64.AppImage.sha256`
- `ppmc-$TAG-linux-x86_64.deb`
- `SHA256SUMS`

### 18.2 PDFium 同梱

リリース成果物には PDFium の動的ライブラリを同梱する方針とする。アプリは実行ディレクトリの PDFium ライブラリを優先し、開発時はシステムライブラリにフォールバックできるようにする。一般ユーザーには追加セットアップを要求しない。

### 18.3 リリース自動化

将来的に GitHub Actions で以下を自動化する。

- Linux 向け AppImage / deb のビルド。
- PDFium ライブラリの同梱。
- チェックサム生成。
- GitHub Releases への成果物アップロード。
- インストールスクリプトの更新。
- `ppmc-$TAG-linux-x86_64.AppImage` と `ppmc-$TAG-linux-x86_64.deb` の release asset 生成。

## 20. 初期決定案

現時点の初期決定案は以下とする。

- MVP は Linux デスクトップ向け。
- 実装は Rust stable。
- GUI は Tauri を第一候補。
- PDF レンダリングは `pdfium-render` を第一候補。
- PDF 表示は WebView 組み込みビューアに任せず、Rust 側で画像化して表示する。
- リリース配布では PDFium 同梱を第一候補とする。
- 設定ファイルは TOML。
- ノートは外部 Markdown ファイルから始める。
- モバイル連携はスマートフォンのブラウザで動く Web コンソール方式とする。
- リモート操作は MVP の主要機能として扱う。
- MVP のスマートフォン画面は「操作 + 状態表示」を範囲とする。
- ノート表示とサムネイル表示は早期拡張として扱う。
- iOS / Android ネイティブアプリは作らない。
- 初期公開先は GitHub Releases とする。
- Linux 配布は AppImage と deb を優先し、curl/wget 用インストールスクリプトを用意する。
- インストール後は `ppmc` コマンドで起動できる状態を目標にする。
- snap は後続対応候補とする。
